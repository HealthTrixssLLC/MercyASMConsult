ASM Consultant Summary
June 12, 2026 – ASM Process Review
Executive Takeaway

The June 12 discussion moved from future-state architecture into a detailed working review of the actual current-state UHC SQL/query logic, payer file availability, MAO002/MAO004 reconciliation, and known operational gaps.

The main themes were:

Mercy does not receive MAO002/MAO004 consistently across all payers.
UHC current-state logic includes payer-specific filtering, formatting, and manual correction steps.
Technical specifications / payer layouts are not consistently governed or refreshed.
MAO002 analysis for Blue Cross of Oklahoma surfaced a potential reconciliation gap.
The team agreed that volume completeness and payer-response reconciliation need to become formalized controls.
1. Response File Availability Is Incomplete Across Payers

Samir began by trying to understand which files Mercy had available for reconciliation. Sandra clarified that the only MAO002 files Mercy receives are from Blue Cross of Oklahoma. Jennifer added that the only risk-related files currently received are for:

Aetna
Blue Cross of Oklahoma
Humana
UHC

For Anthem, Arkansas, Essence, and Global Health, Mercy is not receiving those files, which Jennifer described as one of the current pain points. The 2026 mid-year sweep files discussed were confirmed as Mercy-submitted sweep files.

Consultant Observation

This is a significant reconciliation limitation. Mercy cannot fully validate downstream CMS submission or acceptance across all plans unless the plans provide usable response files, ideally including MAO002 and MAO004.

2. Need for MAO File Completeness Analysis

Samir recommended starting with a volume completeness analysis before relying on the MAO files for reconciliation. The idea is to test whether the response files appear complete by comparing record/member/diagnosis volumes by:

Health plan
Date of service month
File received month
Submission period
Record counts

He gave an example from prior experience where MAO files appeared usable, but a monthly distribution revealed missing months. The concern is that if Mercy builds reconciliation logic on incomplete MAO data, the process could produce false gaps.

Recommended Framing

Before using MAO002/MAO004 for operational conclusions, Mercy should establish:

Which files were received
Which files were loaded
What months/DOS periods are represented
Whether volume distribution looks reasonable
Whether missing volumes reflect payer filtering or missing files
3. Review of UHC SQL Logic

The team then shifted into Linda's UHC query logic. Samir confirmed that he was reviewing the UHC queries Linda had sent. Linda explained that UHC requires separate institutional and ambulatory submissions, with different source tables and different file layouts. UHC also has the 40-diagnosis-code-per-line limitation, requiring the extract to split diagnosis codes into DX 1–40 and DX 41–80 when needed.

Key UHC-Specific Items Discussed
Separate institutional and ambulatory extracts.
Payer-specific layout requirements.
UHC 40 diagnosis code limit per line.
Use of CMS-allowable CPT list filtering.
Use of bill type values such as 111 and 131.
Service line handling to avoid duplicate-looking rows.
Payer-driven expectations that may not align cleanly with CMS-native processing logic.
4. CPT Filtering: Advisory Concern vs Current Payer Requirement

Samir questioned whether Mercy should be filtering CPT codes at the medical group/provider side, noting that conceptually this filtering is typically done by the health plan or CMS downstream. Linda explained that payers have asked Mercy to filter to CMS-allowable CPT codes and that non-matching CPTs have historically generated rejections. For UHC specifically, Linda indicated the rejection occurs at the claim-line level rather than necessarily rejecting the entire file.

Sandra asked the practical question: if CMS and the payer would reject the records anyway, what would be the value of sending them? Samir clarified that his experience was from a health-plan context where broader data could be useful for suspecting, but he acknowledged that based on Linda's payer instructions, Mercy is currently expected to filter those records out.

Consultant Observation

This is not necessarily a “wrong” current-state practice. The finding is more nuanced:

Mercy is performing payer-requested filtering, but the exact payer requirement should be documented, versioned, and periodically revalidated because filtering at the provider side creates risk if the wrong CPT value set, effective date, or interpretation is used.

5. Technical Specs / Data Dictionaries Are a Governance Gap

A major thread was the difference between:

A payer template
A technical specification
A data dictionary
Field-level required/optional logic
Current payer instructions

Linda stated that Mercy has payer templates showing the fields they want, but not necessarily full data dictionaries. Kate and Jay pushed for the current payer technical specifications to be requested so Mercy can confirm whether long-standing assumptions are still valid. Linda said she had recently asked payers whether any changes were needed, and payers responded that no changes were required; however, Kate clarified that the ask should be for the current technical specs themselves, not just whether changes exist.

Consultant Observation

This is a strong finding for the assessment:

Mercy has operational knowledge and payer templates, but the process would benefit from a governed payer-spec repository with current versions, effective dates, source emails, owner, and sign-off.

6. UHC Service Line / Duplicate Handling

Samir questioned why the UHC logic focuses on service line one and whether excluding other service lines could unintentionally exclude a valid face-to-face encounter. Linda explained that if multiple allowable CPTs come through with the same diagnosis codes, the file can look duplicative to UHC, and the payer wants one CPT/revenue-code/diagnosis combination rather than multiple rows that repeat the same diagnosis set. Kate raised a related concern: if CPT1 and CPT3 are both CMS-valid and have slightly different diagnosis associations, filtering to one could potentially remove useful information.

Consultant Observation

This should be documented as an item to validate with UHC:

Is Mercy expected to send only one CPT per diagnosis line?
If multiple CPTs are valid, which one should be selected?
How should diagnosis differences across service lines be handled?
Is this UHC-only or applicable across other payers?
7. Bill Type Mapping and Loss of Fidelity

Jay questioned the bill type mapping, specifically whether all institutional records should be mapped into 111 or 131. Linda explained that for UHC, 111 is used when the base class is 1, and everything else is sent as 131 because that is what UHC requested.

Jay's concern was that defaulting or collapsing bill types can lose fidelity around facility type, frequency, interim vs final billing, and other information that CMS filtering logic may use. The team discussed that UHC may be linking the supplemental file back to the original claim and retrieving more precise bill type information from the original claim rather than using the ASM field as a pass-through.

Consultant Observation

This is another payer-validation item rather than an immediate defect:

If UHC links ASM records back to original claims, simplified bill type mapping may be acceptable. If UHC submits unlinked records using Mercy's mapped values, then fidelity loss could matter.

8. Manual Correction and Error Handling

Samir asked how rejected UHC records are handled. Linda explained that UHC sends an error file containing only the rejected records. For correctable errors, she manually reviews the file, makes corrections, and sends the corrected file back. Common examples included:

Last name mismatch
MBI issues
Member not found in plan

For some issues, UHC may instruct Mercy to blank out a field and allow UHC to populate it. For member-not-found issues, Linda checks Epic/e-verification and may escalate the issue to Mercy/UHC relationship owners if Mercy shows the member as valid.

Consultant Observation

This is a clear opportunity for future-state controls:

Capture error file intake systematically.
Store rejected records.
Categorize fixable vs non-fixable errors.
Track correction status.
Preserve original and corrected values.
Link corrected resubmissions back to the original Run ID / submission file.

Jay paused to confirm whether the team should continue reviewing current-state Oracle code if Linda is already migrating and improving the process in IDP/Databricks. Sandra confirmed that it still makes sense to review current state because if Linda has already addressed an issue in the new process, that can simply be marked as resolved.

10. Blue Cross of Oklahoma MAO002 Use Case

Later in the meeting, Jennifer walked through her preliminary Blue Cross of Oklahoma MAO002 analysis. She ingested the MAO002 files and reviewed fields such as:

Accepted indicators
Error codes
Initiative type

She observed that “initiative type” included values such as chart review, claims, Part B, IHA, and SBS, but there was no ASM category. Jay clarified that initiative type is not a standard MAO002 field; it is likely added by the health plan.

Jennifer then joined the sweep files to MAO002 and found a concerning scenario: for the 2026 mid-year submission, there were records that were attributed and had HCCs indicated, but were not found in MAO002. The discussion referenced roughly 4,800 diagnosis-code-level records missing from MAO002. Jay framed this as a gap, but not yet proof of root cause: the plan may not have submitted them to CMS, Mercy may not have received the relevant MAO002 file, or the files may not have been fully loaded.

Consultant Observation

This is one of the strongest concrete examples from the session.

The Blue Cross of Oklahoma analysis provides a real use case for building the future reconciliation process:

Start with submitted sweep records.
Identify records expected to appear in MAO002.
Isolate missing records.
Confirm all MAO002 files were received and loaded.
If Mercy data is complete, send sample/member-DOS-diagnosis details to the plan for explanation.
11. MAO002 vs MOR Clarification

Jennifer initially thought that members without 12 months of CMS data might not appear in MAO002. Samir clarified that the 12-month logic is relevant to MOR/payment modeling, not MAO002 encounter reconciliation. Jay summarized the issue: Jennifer's observation is a submission/reconciliation gap, meaning either the plan did not submit the data to CMS, or Mercy does not have the MAO002 file containing it.

Consultant Observation

This distinction is important for the deck:

MAO002 / MAO004 = encounter submission and acceptance reconciliation.
MOR = payment/model reconciliation.
New enrollee logic affects payment/MOR interpretation, not whether a submitted encounter should appear in MAO002.
Key Findings from June 12
Finding 1 — Response file receipt is inconsistent by payer

Mercy receives MAO004 for some payers and MAO002 only for Blue Cross of Oklahoma. Several payers provide no files, creating a material blind spot in reconciliation.

Finding 2 — Payer requirements are driving current-state logic, but supporting specs are not fully governed

Linda's logic is based on long-standing payer instructions, but the team identified a need to request and retain current payer specs to confirm interpretations.

Finding 3 — UHC logic contains payer-specific assumptions requiring validation

Examples include CPT filtering, single CPT/service-line handling, bill type mapping, and legacy fields such as risk assessment code or specialty mapping.

Finding 4 — Manual correction exists for UHC error files

Linda manually reviews UHC error files and corrects fixable issues. This works operationally today but should be controlled and tracked in the future-state process.

Finding 5 — Blue Cross MAO002 analysis surfaced a real reconciliation gap

Jennifer identified submitted records with attribution/HCC indicators that were not found in MAO002, creating a strong test case for payer-response reconciliation and completeness controls.

Recommended Next Steps
Perform MAO file completeness analysis by payer, file month, DOS month, and record/member/diagnosis counts.
Request current technical specs from all payers, not just confirmation that “nothing changed.”
Validate UHC-specific assumptions around CPT filtering, service-line handling, bill type mapping, and legacy fields.
Request SQL/query logic for the next largest payer after UHC, with population-size breakdown to prioritize review.
Use Blue Cross of Oklahoma as the first reconciliation pilot, especially the missing MAO002 records Jennifer found.
Document current vs future-state remediation status so current-state findings that Linda has already resolved in IDP can be marked as closed or addressed.
Design a formal error-file intake and correction workflow for UHC and other payers that provide rejection/error files.
Consultant Assessment

The June 12 discussion was the first session where the team moved from conceptual architecture into detailed operating controls. It exposed a practical truth: Mercy's ASM process is functioning, but it depends heavily on payer-specific instructions, embedded SQL logic, manual review, and incomplete response-file availability.

For the assessment deck, I would treat June 12 as support for three core recommendations:

Govern payer specifications and value sets.
Formalize run/submission/response reconciliation.
Move payer-specific rules into controlled configuration rather than embedded SQL logic.
