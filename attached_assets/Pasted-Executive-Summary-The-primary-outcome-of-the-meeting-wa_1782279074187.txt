Executive Summary

The primary outcome of the meeting was alignment that the engagement is not a traditional current-state assessment. Mercy is actively migrating its ASM/Sweeps process from Oracle to Azure/Databricks and simultaneously implementing a new validation framework. As a result, the assessment must evaluate both the current operating process and the intended future-state architecture, rather than focusing solely on historical operations.

The team also confirmed that the review should encompass all participating Medicare Advantage health plans, not just a single payer, to ensure payer-specific compliance and alignment across the entire sweeps process.

Key Findings
1. Scope Expanded Beyond "Current State"

Sandra repeatedly emphasized that Mercy is mid-migration:

Oracle → Azure/Databricks
Legacy extraction process → redesigned architecture
No formal validation process → new validation framework

This means discovery must examine:

Current State
Existing sweeps process
Existing file layouts
Existing payer submissions
Target State
Databricks implementation
Curated/semantic data layer
New validation process
Future operational model

Consulting implication:
The engagement should be positioned as a Current State + Transitional Future State Assessment.

This aligns very closely with the architecture discussions we've had regarding:

Clarity normalized sources
Curated data layer
Generalized ASM engine
Validation layer
Reverse reconciliation process
2. Oracle Environment Is Out of Scope

A major clarification:

Mercy does not want HealthTrix to evaluate the historical Oracle implementation.

Instead:

Review Databricks notebooks
Review migrated SQL logic
Review future-state design documentation
Review current output files

Sandra explicitly stated that the Oracle implementation is no longer the target environment.

Risk

This creates a potential blind spot:

If migration introduced transformation issues, logic changes, or data loss, those issues may not be fully visible.

Samir appropriately raised this concern.

Finding:
Migration validation risk exists because source-to-target conversion is outside assessment scope.
